# VineTrack Paddock Geometry Writer Spec

Canonical format and algorithm used by the iOS app to create / update paddock
("block") geometry. This is the contract any external writer (Lovable web
portal, migration scripts, future backends) MUST follow so paddocks created on
the web sync cleanly into the iOS app.

> **Status: documentation only.** No code or schema is changed by this file.
> Source of truth is the Swift code referenced in section 8.

Companion docs:
- `docs/supabase-schema.md` — Supabase column inventory.
- `sql/005_paddocks_sync.sql`, `sql/020_paddocks_intermediate_post_spacing.sql`
  — DDL for the `paddocks` table.

## 1. `polygon_points` canonical JSON shape

Stored in `paddocks.polygon_points` (JSONB). Decoded into Swift
`[CoordinatePoint]` via `Paddock.CodingKeys.polygonPoints`.

### Element shape

Array of objects. Each object MUST contain:

| Key | Type | Required | Notes |
| --- | --- | --- | --- |
| `id` | string (UUID v4) | Yes | Stable per-vertex identifier. Allows the iOS boundary editor to track moved vs inserted points. Generate fresh UUIDs for new vertices. Persist them — do not regenerate on every save. |
| `latitude` | number (double, decimal degrees, WGS84) | Yes | |
| `longitude` | number (double, decimal degrees, WGS84) | Yes | |

```json
[
  { "id": "5C2E…", "latitude": -34.51234, "longitude": 138.71234 },
  { "id": "9A11…", "latitude": -34.51240, "longitude": 138.71300 },
  { "id": "B7F3…", "latitude": -34.51280, "longitude": 138.71280 }
]
```

### Order, closure, precision

- **Coordinate order:** stored as separate `latitude` / `longitude` keys, not a
  GeoJSON `[lon, lat]` tuple. Do NOT emit GeoJSON shape into `polygon_points`.
- **Winding:** not enforced. iOS area calc uses `abs(...)` shoelace, so
  CW or CCW both work. Self-intersecting polygons are NOT handled — see §6.
- **Closure:** polygon is **open** in storage. The first vertex is **NOT**
  repeated as the last vertex. iOS closes the ring implicitly via
  `j = (i + 1) % n` in `Paddock.areaHectares`, `RowGeometry.clipLineToPolygon`,
  and `BoundaryMapEditor`.
- **Precision:** doubles. iOS keeps full Core Location precision; ~7 decimal
  places (≈1 cm) is more than enough. No rounding is required, and no rounding
  is applied on save.
- **Minimum length:** see §6.

## 2. `rows` canonical JSON shape

Stored in `paddocks.rows` (JSONB). Decoded into Swift `[PaddockRow]`.

Each row is the straight-line trellis row generated by §4. Rows are simple
two-point line segments (start → end), NOT polylines.

### Element shape

| Key | Type | Required | Notes |
| --- | --- | --- | --- |
| `id` | string (UUID) | Yes | Stable per-row id. |
| `number` | integer | Yes | The user-visible row number (e.g. 1..N). May start at any positive integer (`rowStartNumber`, default `1`) and may run ascending OR descending across the field — see §4. |
| `startPoint` | object (`CoordinatePoint`) | Yes | `{ id, latitude, longitude }` — same shape as §1. |
| `endPoint` | object (`CoordinatePoint`) | Yes | Same shape. |

```json
[
  {
    "id": "F0A2…",
    "number": 1,
    "startPoint": { "id": "…", "latitude": -34.5121, "longitude": 138.7128 },
    "endPoint":   { "id": "…", "latitude": -34.5126, "longitude": 138.7131 }
  }
]
```

### Notes / non-fields

- There is **no** `length`, `lengthMeters`, `active`, `skipped`, or `polyline`
  field in the iOS model. Adding extra keys is harmless on decode (Swift
  ignores unknown keys), but they will not survive a round-trip and must not
  be relied on.
- Row geometry is always a single straight segment. Curved or polyline rows
  are not currently supported by the data model.
- Row length is always **derived** from `startPoint` / `endPoint` (see §5).
- Row count is `rows.length`. There is no separate `rowCount` column.

## 3. `variety_allocations` canonical JSON shape

Stored in `paddocks.variety_allocations` (JSONB). Decoded into Swift
`[PaddockVarietyAllocation]`.

### Element shape

| Key | Type | Required | Notes |
| --- | --- | --- | --- |
| `id` | string (UUID) | Yes (defaults to a new UUID on decode if missing, but always emit one) | Stable allocation row id. |
| `varietyId` | string (UUID) | Yes | FK to a `GrapeVariety.id` belonging to the same vineyard. The web writer MUST pick from `grape_varieties` for that vineyard, not invent ids. |
| `percent` | number (double, 0–100) | Yes | Percentage of the block planted to this variety. Allocations within a paddock should sum to ≈100, but iOS does NOT reject other totals; it warns the user. |

```json
[
  { "id": "…", "varietyId": "…", "percent": 60 },
  { "id": "…", "varietyId": "…", "percent": 40 }
]
```

### Not stored at allocation level

The current model intentionally does **not** include:

- `clone`
- `rootstock`
- `plantingYear` (paddock-level only — `paddocks.planting_year`)
- `polygon` / sub-block geometry
- `rowIds` / `rowRange`

If a future writer wants to express clone/rootstock/sub-block geometry, that
must be a schema extension agreed with the iOS team — do NOT smuggle extra
keys into `variety_allocations` in MVP, because the iOS Swift type is a fixed
shape.

Optional fields are simply **omitted** from the JSON. Encoding them as `null`
also decodes correctly (Swift uses `decodeIfPresent`), but omission is
preferred to keep payloads small.

## 4. Row generation algorithm

Reference implementation: `RowGeometry.swift::calculateRowLines(...)` plus the
caller in `EditPaddockSheet.swift::savePaddock()` (lines ~778–804).

Inputs (paddock-level):

| Field | Type | Notes |
| --- | --- | --- |
| `polygon_points` | `[CoordinatePoint]` | Open polygon, ≥3 vertices. |
| `row_direction` | double (degrees) | 0…360. Compass bearing of the row direction (0 = North, 90 = East). |
| `row_width` | double (metres) | Distance between adjacent rows, perpendicular to row direction. iOS UI clamps to 0.0…4.0 m. |
| `row_offset` | double (metres) | Lateral shift of the whole row pattern perpendicular to row direction. iOS UI clamps to ±50 m. |
| `rows.length` (count) | int | Number of rows requested. |
| `vine_spacing` | double (metres) | Used by §5, not by row line projection. |
| `rowStartNumber` (UI only) | int | Number assigned to the first row. NOT stored as its own column — encoded into `rows[*].number`. Default 1. |
| `rowNumberAscending` (UI only) | bool | Whether `rowStartNumber` is on the "left" side of the bearing (true) or the "right" (false). Encoded into `rows[*].number`. |

### Algorithm

1. **Guard.** Return `[]` if `polygon.count < 3`, `count <= 0`, or `width <= 0`.
2. **Centroid (planar).** Compute the arithmetic mean of polygon latitudes /
   longitudes. This is the local origin used for all subsequent metric
   conversions.
3. **Local metric scaling.** Use a flat-earth / equirectangular approximation
   anchored at the centroid:
   - `mPerDegLat = 111320.0`
   - `mPerDegLon = 111320.0 * cos(centroidLat * π / 180)`
   This is accurate enough for paddock-scale (≤ a few km) work and matches
   what iOS uses for area, row length, and post counts.
4. **Bearing.** Convert `row_direction` (degrees) to radians: `bearingRad`.
   The perpendicular axis is `perpRad = bearingRad + π/2`.
   Note: in iOS the bearing is encoded as `(latΔ ∝ cos, lonΔ ∝ sin)`,
   matching `north = 0°, east = 90°` only when latitude/longitude are treated
   as a planar (lat, lon) Cartesian frame using the metre conversions above.
   The web writer MUST follow the same convention to produce identical lines.
5. **Half length.** Find the maximum pairwise distance between polygon
   vertices in metres (`maxDist`). Use `halfLen = maxDist * 1.5`. This guarantees
   each candidate line is long enough to fully cross the polygon before clipping.
6. **Row offsets.** Distribute rows symmetrically around the centroid:
   - `totalW = (count - 1) * width`
   - `startOff = -totalW / 2`
   - For row `i ∈ [0, count)`: `off = startOff + i*width + row_offset`
7. **Candidate line per row.** Centre point of row `i`:
   - `cLat = centroidLat + off * cos(perpRad) / mPerDegLat`
   - `cLon = centroidLon + off * sin(perpRad) / mPerDegLon`
   Endpoints (before clipping):
   - `dLat = halfLen * cos(bearingRad) / mPerDegLat`
   - `dLon = halfLen * sin(bearingRad) / mPerDegLon`
   - `start = (cLat - dLat, cLon - dLon)`
   - `end   = (cLat + dLat, cLon + dLon)`
8. **Clip to polygon.** Intersect the candidate line with every polygon edge.
   - If fewer than 2 intersections → drop the row entirely (no `PaddockRow`
     emitted, and the row index is skipped — see step 9).
   - If ≥2 intersections → sort intersections along the candidate line
     parameter and use the first and last as the clipped `start` / `end`.
   Convex polygons always give exactly 2 intersections; concave polygons may
   give more, and only the outermost pair is kept (i.e. the line is treated as
   a single segment, not split around concavities).
9. **Row numbering.** After `calculateRowLines` returns the geometric lines
   (in centroid-perpendicular order, i.e. `i = 0` is the "left-most" row in
   the `perpRad` direction), iOS emits exactly `count` rows in the same order.
   The `number` field is filled by:
   ```
   number = ascending ? rowStartNumber + i
                      : rowStartNumber + (count - 1 - i)
   ```
   Rows that were dropped by clipping (step 8) currently still consume an
   index in the iOS save loop and get a `(0,0)`–`(0,0)` placeholder
   (`EditPaddockSheet.savePaddock`, lines ~795–798). The web writer SHOULD
   instead omit dropped rows entirely; iOS will read them back fine because
   the count comes from `rows.length`.

### Sort order on disk

`rows` is stored in **left-to-right order along `perpRad`**, not by `number`.
Consumers that need rows in user-numbered order MUST sort by `number`
themselves.

## 5. Derived calculations

All derived values are computed on read in `Paddock` (Swift extension in
`Paddock.swift`, lines 129–239). The web portal MUST NOT persist derived
values into base columns — store inputs only. Display-side mirror formulae:

### 5.1 Area (hectares) — `areaHectares`

Shoelace on equirectangular-projected metres around the polygon centroid:

```
mPerDegLat = 111320
mPerDegLon = 111320 * cos(centroidLat * π / 180)
areaSqM    = abs( Σ (xi*yj - xj*yi) ) / 2     (i, j=(i+1)%n; xi=lon*mPerDegLon, yi=lat*mPerDegLat)
hectares   = areaSqM / 10000
```

### 5.2 Total row length (metres) — `totalRowLengthMetres`

```
Σ over rows: hypot( (end.lat-start.lat)*mPerDegLat,
                    (end.lon-start.lon)*mPerDegLon )
```

`effectiveTotalRowLength = rowLengthOverride ?? totalRowLengthMetres`

### 5.3 Vine count — `effectiveVineCount`

```
estimatedVineCount = floor(effectiveTotalRowLength / vineSpacing)   (vineSpacing > 0)
effectiveVineCount = vineCountOverride ?? estimatedVineCount
```

### 5.4 Intermediate posts — `intermediatePostCount`

Counts only "internal" trellis posts; the two end posts of every row are
excluded:

```
rawPosts = floor(effectiveTotalRowLength / intermediatePostSpacing)
endPosts = 2 * rows.count
intermediatePostCount = max(0, rawPosts - endPosts)
```

Returns `nil` if `intermediatePostSpacing` is missing/≤0 or row length is 0.

### 5.5 Total emitters — `totalEmitters`

```
totalEmitters = floor(effectiveTotalRowLength / emitterSpacing)
```

### 5.6 Irrigation rates

```
emittersPerHa  = 10000 / (rowWidth * emitterSpacing)
litresPerHaHr  = emittersPerHa * flowPerEmitter            (L/ha/hr)
mlPerHaHr      = litresPerHaHr / 1_000_000                 (ML/ha/hr)
mmPerHr        = mlPerHaHr * 100                           (mm/hr)

litresPerHour  = (effectiveTotalRowLength / emitterSpacing) * flowPerEmitter
litresPerVineHr = (vineSpacing / emitterSpacing) * flowPerEmitter
```

All return `nil` unless `emitterSpacing > 0`, `flowPerEmitter > 0`, and
`rowWidth > 0` where applicable.

## 6. Validation rules

The iOS app is mostly permissive — it shows warnings rather than rejecting
saves — but the web writer should match these effective rules to avoid
producing paddocks iOS can't render correctly.

| Field | Rule | Source |
| --- | --- | --- |
| `polygon_points.length` | ≥ 3 to render area / rows / map polygon. iOS guards on `count >= 3` everywhere. | `Paddock.areaHectares`, `RowGeometry.calculateRowLines`, `BoundaryMapEditor` |
| Self-intersection | Not detected. Avoid bow-tie polygons; row clipping degrades. | implicit |
| `row_direction` | 0 ≤ x ≤ 360 (UI slider). Values outside that range still work mathematically but should be normalised. | `EditPaddockSheet` slider 0…360 |
| `row_width` | > 0. UI slider 0.0…4.0 m. Hard floor for row generation: > 0 (else no rows). | `RowGeometry.calculateRowLines` |
| `row_offset` | Any double; UI clamps -50…50 m. | `EditPaddockSheet` |
| `vine_spacing` | > 0 for vine count. UI slider 0.5…3.0 m, default 1.0. | `Paddock.estimatedVineCount` |
| `emitter_spacing` | > 0 for irrigation calcs; otherwise nil. | `Paddock.litresPerHaPerHour` etc. |
| `flow_per_emitter` | > 0 for irrigation calcs. | same |
| `intermediate_post_spacing` | > 0 for post count; otherwise nil. | `Paddock.intermediatePostCount` |
| `rows[*].number` | Positive integer expected. iOS only uses it for display + ordering. | `EditPaddockSheet` |

Defaults applied by the iOS decoder when a column is missing:

| Field | Default |
| --- | --- |
| `row_width` | 2.5 |
| `row_offset` | 0 |
| `vine_spacing` | 1.0 |
| `variety_allocations` | `[]` |
| `polygon_points`, `rows` | required by decoder — must be present (can be `[]`) |

Optional/nullable: `vine_count_override`, `row_length_override`,
`flow_per_emitter`, `emitter_spacing`, `intermediate_post_spacing`,
all phenology dates, `planting_year`, `calculation_mode_override`,
`reset_mode_override`.

## 7. Sync requirements (web → iOS round-trip)

For a paddock created or updated by the web portal to sync safely into the
iOS app, the upsert payload MUST include the following columns. Names are
the snake_case Supabase column names (matching `BackendPaddockUpsert`
`CodingKeys`). Optional columns may be omitted or sent as `null`.

### Required

| Column | Type | Notes |
| --- | --- | --- |
| `id` | uuid | Stable, generated client-side. Conflict key on upsert. |
| `vineyard_id` | uuid | Must be a vineyard the writer's user has the right role on. RLS enforces. |
| `name` | text | Non-empty. iOS save button is disabled when empty. |
| `polygon_points` | jsonb | Per §1. Empty array allowed during draft, but rendering / area / row generation require ≥3. |
| `rows` | jsonb | Per §2. Empty array allowed. |
| `row_direction` | double | Required by iOS decoder. Use 0 if unknown. |
| `client_updated_at` | timestamptz (ISO 8601) | Used by iOS conflict resolution (`applyRemote` last-write-wins). MUST reflect the user-edit time, not the DB write time. |

### Strongly recommended

| Column | Type | Notes |
| --- | --- | --- |
| `row_width` | double | Defaults to 2.5 if missing — surprises users. |
| `row_offset` | double | Defaults to 0. |
| `vine_spacing` | double | Defaults to 1.0 — affects vine count. |
| `variety_allocations` | jsonb | Per §3. Default `[]`. |
| `created_by` | uuid | `auth.users.id` of the creating user. iOS sets this to `auth.userId` on first push. |

### Optional

`vine_count_override`, `row_length_override`, `flow_per_emitter`,
`emitter_spacing`, `intermediate_post_spacing`, `budburst_date`,
`flowering_date`, `veraison_date`, `harvest_date`, `planting_year`,
`calculation_mode_override` (`"baseTen" | "huglin" | "winkler"` — see
`GDDCalculationMode`), `reset_mode_override`
(`"seasonStart" | "budburst" | "flowering" | "veraison"`).

### Server-managed — DO NOT SEND from a client writer

These are filled by Postgres triggers / RLS / RPCs and the iOS upsert
payload deliberately excludes them. The web writer must do the same.

- `created_at` (default `now()`)
- `updated_at` (trigger `paddocks_set_updated_at`)
- `updated_by` (set by RPC / trigger)
- `deleted_at` (only via `soft_delete_paddock(p_paddock_id uuid)` RPC,
  which checks `owner | manager | supervisor`)
- `sync_version` (default 1; reserved)

### Conflict key

`upsert(..., onConflict: "id")` — the `id` column is the merge key. Any web
writer MUST use the same id when updating an existing paddock; do not insert
a new row.

### Soft delete

Web writers that need to delete must call the
`soft_delete_paddock(p_paddock_id uuid)` RPC. Direct `DELETE` is blocked by
RLS (`paddocks_no_client_hard_delete`).

## 9. Web portal write contract — checklist

Before enabling block create/edit from Lovable:

1. Use the same Supabase project + anon key (RLS-enforced).
2. Build the upsert payload with the exact JSON shapes in §1, §2, §3.
3. Always send all columns in §7 "Required" + populate `client_updated_at`
   with the user's edit time in ISO 8601.
4. Generate a fresh `id` (uuid v4) for new paddocks; reuse existing `id` on
   edit.
5. Generate fresh per-vertex / per-row / per-allocation `id`s on creation
   and persist them on subsequent edits.
6. Run the §4 algorithm exactly as specified — same centroid, same
   `mPerDegLat / mPerDegLon`, same `bearingRad / perpRad` convention,
   same `halfLen = 1.5 * maxPairwiseDistance`. Producing rows with a
   different convention will make iOS render rows that don't match the
   user's web preview after sync.
7. Never write `created_at`, `updated_at`, `updated_by`, `deleted_at`,
   `sync_version` from the web client.
8. For deletes, call `soft_delete_paddock` RPC. Never `DELETE`.
9. Do not store derived values (`area`, `vineCount`, `mmPerHr`, …) in any
   column. They are recomputed on every read.
