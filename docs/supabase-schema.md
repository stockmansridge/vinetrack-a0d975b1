# VineTrack Supabase Schema Reference

Canonical schema reference for the iOS app and any future web portal work
(Lovable read-only admin portal). Generated by auditing `sql/001` through
`sql/021` plus the iOS Backend DTOs in `ios/VineTrackV2/Backend/`.

> **Do not run migrations** based on this file. It documents the live schema
> the iOS app already writes to. Schema changes must go through a numbered
> file in `sql/` and be reviewed.

---

## 1. Project / connection

The iOS app reads the Supabase project URL and anon key at runtime from
`AppConfig` (`ios/VineTrackV2/App/AppConfig.swift`):

| Source | Public env var | Notes |
| --- | --- | --- |
| iOS app | `EXPO_PUBLIC_SUPABASE_URL` | URL of the live Supabase project |
| iOS app | `EXPO_PUBLIC_SUPABASE_ANON_KEY` | Anonymous (RLS-enforced) key |

The same project is used by edge functions in `supabase/functions/`
(`davis-proxy`, etc.). Lovable should connect to the same project ref

## 2. Conventions

All operational tables follow the same pattern:

- **Primary key**: `id uuid` (`gen_random_uuid()`).
- **Tenant key**: `vineyard_id uuid not null references vineyards(id) on delete cascade`.
- **Sync columns**:
  - `created_at timestamptz not null default now()`
  - `updated_at timestamptz not null default now()` (auto-bumped by `set_updated_at()` trigger)
  - `deleted_at timestamptz null` (soft delete)
  - `client_updated_at timestamptz null` (set by client; used as the conflict tiebreaker)
  - `sync_version integer not null default 1`
- **Author columns**: `created_by uuid references auth.users(id)`,
  `updated_by uuid references auth.users(id)`.
- **RLS**: enabled on every table.
  - `SELECT` → any vineyard member (`is_vineyard_member`).
  - `INSERT` / `UPDATE` → role-gated via `has_vineyard_role(...)`.
  - `DELETE` → blocked from clients; soft-delete RPCs perform the update.
- **Soft delete RPCs**: `soft_delete_<table>(p_id uuid)` set `deleted_at = now()`.

Helper functions:

- `is_vineyard_member(p_vineyard_id uuid) → boolean`
- `vineyard_role(p_vineyard_id uuid) → text`
- `has_vineyard_role(p_vineyard_id uuid, allowed_roles text[]) → boolean`
- `vineyard_member_role(p_vineyard_id uuid) → text` (alias used by 021)
- `is_vineyard_owner_or_manager(p_vineyard_id uuid) → boolean`
- `is_admin() → boolean` (admin allowlist)
- `storage_first_folder_uuid(name text) → uuid`

Roles: `'owner' | 'manager' | 'supervisor' | 'operator'`.

### 3.8 `paddocks` ⭐ (most-misunderstood table)

This is the canonical paddock/block setup table. Lovable's narrow probe
missed most of these columns.

| Column | Type | Notes |
| --- | --- | --- |
| `id` | uuid | PK |

| `name` | text | not null |
| `row_direction` | double precision | bearing in degrees |
| `row_width` | double precision | metres between rows |
| `row_offset` | double precision | offset of first row from polygon edge (m) |
| `vine_spacing` | double precision | metres between vines along a row |
| `vine_count_override` | integer | optional manual override |
| `row_length_override` | double precision | optional manual override (m) |
| `flow_per_emitter` | double precision | L/h per dripper |
| `emitter_spacing` | double precision | metres between emitters |
| `intermediate_post_spacing` | double precision | metres between trellis intermediate posts (added in 020) |
| `budburst_date` | timestamptz | phenology |
| `flowering_date` | timestamptz | phenology |
| `veraison_date` | timestamptz | phenology |
| `harvest_date` | timestamptz | phenology |
| `planting_year` | integer | |
| `calculation_mode_override` | text | optional override of vineyard default |
| `reset_mode_override` | text | optional override of vineyard default |
| `polygon_points` | jsonb | array of `{lat, lng}` (the paddock boundary) |
| `rows` | jsonb | array of row geometries (start/end coords, length, etc.) |
| `variety_allocations` | jsonb | per-block variety/clone/rootstock allocations |
| `created_by` / `updated_by` | uuid | |
| `created_at` / `updated_at` / `deleted_at` | timestamptz | |
| `client_updated_at` | timestamptz | |
| `sync_version` | integer | |

#### JSONB shapes

`polygon_points`: ordered ring of vertices.

```json
[
  { "latitude": -34.9112, "longitude": 138.6711 },
  { "latitude": -34.9114, "longitude": 138.6720 },
  ...
]
```

`rows`: per-row geometry (one entry per planted row).

```json
[
  {
    "id": "uuid",
    "rowNumber": 1,
    "start": { "latitude": -34.91, "longitude": 138.67 },
    "end":   { "latitude": -34.91, "longitude": 138.67 },
    "length": 142.3,
    "isSkipped": false
  },
  ...
]
```

`variety_allocations`: blocks within the paddock, each with its own variety
and a polygon (subset of the paddock polygon).

```json
[
  {
    "id": "uuid",
    "variety": "Shiraz",
    "clone": "BVRC30",
    "rootstock": "1103P",
    "plantingYear": 2008,
    "polygonPoints": [ { "latitude": ..., "longitude": ... } ],
    "rowIds": ["uuid", "uuid"]
  }
]
```

#### Computed (not stored)

The iOS app derives several values rather than storing them, so the same
inputs reach all devices:

| Derived value | Source |
| --- | --- |
| Area (ha) | from `polygon_points` (planar projection) |
| Row count | `rows.length` |
| Row length (avg/total) | from `rows[*].length` (or `row_length_override`) |
| Vine count | `sum(rows[*].length / vine_spacing)` unless `vine_count_override` is set |
| Intermediate post count | from `rows[*].length / intermediate_post_spacing` |
| Emitter count | from `rows[*].length / emitter_spacing` |

A web portal that wants to show paddock metrics should compute them the
same way from the JSONB columns rather than expecting precomputed columns.

RLS:
- SELECT → members.
- INSERT/UPDATE → all four operational roles.
- Soft delete via `soft_delete_paddock(id)` (owner/manager/supervisor).

iOS DTO: `BackendPaddock`. Repository: `SupabasePaddockSyncRepository`.

### 3.17 `vineyard_weather_integrations` (sql/021) — restricted

Stores per-vineyard weather provider credentials (Davis WeatherLink,
Wunderground).

| Column | Type | Notes |
| --- | --- | --- |
| `id` | uuid | PK |
| `vineyard_id` | uuid | not null |
| `provider` | text | check: `davis_weatherlink` \| `wunderground` |
| `api_key` | text | sensitive |
| `api_secret` | text | secret — never returned to clients |
| `station_id`, `station_name` | text | |
| `station_latitude`, `station_longitude` | double precision | |
| `has_leaf_wetness`, `has_rain`, `has_wind`, `has_temperature_humidity` | boolean | |
| `detected_sensors` | text[] | |
| `configured_by` | uuid | FK → `auth.users(id)` |
| `updated_at`, `last_tested_at` | timestamptz | |
| `last_test_status` | text | |
| `is_active` | boolean | default true |

Unique `(vineyard_id, provider)`.

> **RLS is locked. There are no policies on this table. All client SELECT/
> INSERT/UPDATE/DELETE is revoked.** Access goes only through:
>
> - `get_vineyard_weather_integration(vineyard_id, provider)` — members
>   read non-secret fields only (returns `has_api_key` / `has_api_secret`
>   booleans, never the values).
> - `save_vineyard_weather_integration(...)` — owner/manager.
> - `delete_vineyard_weather_integration(...)` — owner/manager.
> - `reveal_vineyard_weather_integration_credentials(...)` — owner/manager
>   for the management UI.
>
> The edge function `davis-proxy` reads `api_secret` server-side using
> the service role.

**Lovable / web portal must never SELECT this table directly** and must
never call `reveal_vineyard_weather_integration_credentials`. Use
`get_vineyard_weather_integration` only if a status display is needed.

## 7. Intentionally local-only data

The following are **not** synced through Supabase by design:

- **Davis API secrets on the iOS device** — handled via the centralised
  RPCs above; legacy Keychain copies are still read for backward
  compatibility but new writes go through `save_vineyard_weather_integration`.
- **UI preferences / device settings** — theme, last-selected map layer,
  dashboard expansion state, etc. (UserDefaults).
- **Transient sync metadata** — `lastSync` timestamps, recovery flags
  (e.g. `vinetrack_*_sync_reset_v1`) and per-service migration markers
  live in UserDefaults.
- **Caches** — weather snapshots, chemical info lookups, derived
  computed values (area, vine count, post count) — recomputed on demand
  from the synced inputs.

If you find domain data (counts, settings used by the Vineyard Summary,
business records) that is local-only, that is a sync bug, not a design
choice; raise it.

## 8. Lovable / web portal read-only guidance

For the MVP read-only admin / partner portal:

1. **Connect to the same Supabase project** the iOS app uses
   (`EXPO_PUBLIC_SUPABASE_URL`). Use the **anon key only**.
2. **Use the existing tables and RPCs**. Do not propose migrations or
   schema changes from the portal.
3. **No writes** in the MVP. All writes happen from the iOS app, which
   already enforces the right ordering / soft-delete RPCs.
4. **RLS is the source of truth**. The portal authenticates the user
   through Supabase Auth; what they can see is exactly what their
   `vineyard_members` row allows. Do not work around RLS.
5. **Hide columns only because they're not useful in the UI**, not
